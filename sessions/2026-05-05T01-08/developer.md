### A. 헤더 및 요약 (Executive Summary & KPI Focus)
*   **제목:** [분석일: YYYY-MM-DD] SCM 병목 현상 분석 보고서
*   **핵심 메시지 (Alert Box):** (가장 큰 폰트) "Gap 해결 시, 연간 예상 손실 방지액: 5.2억 원 | 매출 증대 기여율: 15% (YOY)"
*   **3대 핵심 지표 (KPI Cards):**
    1.  **현재 예측 정확도 (Forecast Accuracy):** ⚠️ 65% (목표: 85%)
    2.  **잠재적 연간 손실액 (Annual Loss Potential):** 💰 5.2억 원 (최대값 기준)
    3.  **필요 조치 우선순위 (Action Priority):** 🔴 실시간 재고-트렌드 통합 시스템 구축

### B. 문제 정의 및 원인 분석 (Problem Definition)
*   **위젯:** SCM 프로세스 Gap 히트맵
    *   **내용:** 데이터 수집 주기의 불일치 지점(Gap)을 색상으로 표시.
    *   **시각화:** 수요 예측 (Weekly) $\rightarrow$ 생산 계획 (Weekly) $\rightarrow$ 재고 (Daily) 에서 발생하는 데이터 시점 불일치 구간 강조.
*   **요약 텍스트:** "현재 시스템은 판매처별 일일 데이터를 실시간으로 취합하지 못하여, 재고 최적화 결정에 병목 현상이 발생하고 있습니다."

### C. 영향 분석 (Impact Analysis & Quantification)
*   **차트 1: 손실액 구성 분석 (Pie Chart)**
    *   **항목:** 1. 폐기 비용 (Write-off) / 2. 기회 비용 (Opportunity Cost) / 3. 운영 비효율 비용 (Operational Cost)
    *   **목표:** 가장 큰 손실원(기회 비용)을 시각적으로 강조.
*   **차트 2: 시계열 재고 부족/과잉 발생 추이 (Line Chart)**
    *   **X축:** 시간 (월별)
    *   **Y축:** 재고 수준 (단위)
    *   **그래프:** 예측된 최적 재고선과 실제 재고선을 비교하며, Gap 발생 시점의 급격한 하락(품절) 또는 상승(과잉) 지점을 빨간색/파란색으로 표시.

### D. 해결책 제안 및 재무 예측 (Solution & ROI)
*   **위젯 1: 시스템 기능 명세 (Roadmap/Feature List)**
    *   **핵심 기능:** 1. 통합 트렌드 수집 (크롤링), 2. AI 기반 수요 예측, 3. 실시간 재고-발주 연동.
    *   **기술 스택:** Agent/API 기반 자동화 시스템으로 구현.
*   **차트 3: 기대 ROI 시뮬레이션 (Bar Chart)**
    *   **축:** 연도별 (Year 1, Year 2, Year 3)
    *   **막대:** '도입 전 손실액' 대비 '도입 후 예상 이익'을 비교하여, 시스템 도입의 경제적 효과를 직관적으로 제시.

## ⚙️ 2. 백엔드 API 엔드포인트 설계 (RESTful JSON Schema)

대시보드 데이터를 구동하기 위한 메인 엔드포인트를 설계합니다. 모든 분석은 이 단일 엔드포인트를 통해 데이터 로딩 및 제출이 이루어지도록 구성하는 것이 가장 효율적입니다.

### A. 엔드포인트 정의

| 항목 | 값 | 설명 |
| :--- | :--- | :--- |
| **API 이름** | `SCM Analysis Report` | SCM 분석 보고서 데이터를 요청/제출하는 API |
| **URL** | `/api/v1/scm-analysis/report` | |
| **메소드 (GET)** | `GET` | 특정 날짜의 분석 보고서 데이터를 조회합니다. (Read) |
| **메소드 (POST)** | `POST` | 분석 데이터를 전송하고, 분석을 트리거하여 보고서를 생성합니다. (Write/Action) |

### B. 요청 (Request) 스키마

#### 1. GET 요청 파라미터
*   **`date` (string, required):** 분석 기준 날짜 (YYYY-MM-DD). (예: `2026-05-05`)
*   **`user_id` (int, required):** 요청 사용자 ID (권한 검증용).

이 구조는 사용자가 최종 검토 후 '보고서 제출' 버튼을 누를 때 전송되는 모든 분석 데이터를 담습니다.
